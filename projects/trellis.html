Problem
The Computation–Orchestration Boundary

Large language models are increasingly used as interfaces to analytical systems, but live high-stakes financial domains require more than fluent generation: outputs must remain grounded, current, and auditable. The core failure mode is what TRELLIS calls LLM-induced computation — when the model performs numerical calculations directly rather than delegating to deterministic tools, introducing silent errors that compound across multi-step reasoning chains.

TRELLIS enforces a hard architectural boundary: the LLM handles intent understanding, routing, tool planning, and response synthesis. Deterministic pipelines remain the sole source of analytical truth. Unlike prior tool-use systems that assume a static or implicitly consistent world state, TRELLIS explicitly addresses execution-time drift between planning and tool outputs through structured execution semantics.

Architecture
Bounded Orchestration — Two Planes

Orchestration Plane (LLM)

The language model operates exclusively in the orchestration plane: parsing natural language intent, routing to the correct deterministic pipeline, constructing tool call sequences, and synthesising human-readable responses from structured tool outputs. The model never computes numerical values directly.

Execution Plane (Deterministic Pipelines)

All computation — cashflow DCF, bond yield-to-maturity, equity Greeks, FX cross-rates, news sentiment aggregation — runs in deterministic, versioned pipeline modules. Results are returned as structured JSON artifacts with provenance metadata. The orchestration plane consumes but never overrides these artifacts.

MCTS-Guided Routing with LLM-as-Value-Function

Tool selection across the multi-domain treasury benchmark uses Monte Carlo Tree Search where the LLM serves as the value function: scoring candidate tool sequences based on semantic alignment with the query intent. This avoids greedy routing failures on ambiguous cross-domain queries (e.g., a cashflow question that requires FX normalisation).

Temporal Consistency Constraint

Streaming financial data introduces execution-time drift — the market state at planning time may differ from the state when tool calls execute. TRELLIS enforces a temporal consistency constraint: max|τ(z_i) − τ(z_j)| < ε across all tool outputs in a single response, where τ(z) is the timestamp of artifact z. Responses that would violate this bound trigger a consistency-preserving re-query.

Benchmark
Multi-Domain Treasury Coverage

The treasury benchmark covers five financial domains: cashflow (DCF, NPV, IRR, working capital), bonds (YTM, duration, convexity, spread analysis), equities (Greeks, P&L attribution, factor exposure), foreign exchange (spot, forward, cross-rate, carry), and market news (sentiment, event extraction, earnings interpretation). Each domain has a dedicated deterministic pipeline module.

TRELLIS is evaluated against three baselines: Tauric Research Trading Agent (specialist financial agent), FinBot (retrieval-augmented financial QA), and a direct-LLM baseline (GPT-4 with no tool grounding). The temporal consistency constraint and MCTS routing are ablated independently to isolate their contributions.

Evaluation
4-Layer Framework

Layer 1 — Routing Accuracy

Does the orchestrator correctly identify which deterministic pipeline(s) to invoke given a natural language query? Evaluated on 200 held-out queries spanning all five treasury domains.

Layer 2 — Tool Planning

Given correct routing, does the agent construct the right sequence of tool calls with correctly parameterised inputs? Measures parameter extraction accuracy and call-order correctness on multi-step queries.

Layer 3 — Multi-Turn Statefulness

Does the agent maintain coherent state across a multi-turn dialogue? Tests include follow-up questions, context carryover, and correction handling — where the user revises a prior assumption mid-conversation.

Layer 4 — Robustness

Adversarial probing: ambiguous queries designed to elicit direct LLM computation (bypassing the execution plane), conflicting data across streaming sources, and queries that require cross-domain tool composition. TRELLIS shows strong boundary enforcement — LLM-computation bypass rate near zero.
