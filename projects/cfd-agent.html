Problem
Why NL-to-CFD?

Computational fluid dynamics requires deep expertise to set up: choosing the right solver, turbulence model, mesh strategy, and boundary conditions for a given flow regime demands significant domain knowledge. Even experienced engineers spend hours configuring OpenFOAM cases for problems they've solved before.

The goal of this UGRC project was to collapse that workflow — letting engineers describe their problem in plain English ("simulate turbulent flow over a backward-facing step at Re=100") and have an AI agent autonomously handle mesh generation, solver selection, case configuration, execution, and validation. The agent operates as part of a multi-agent scientific platform where it exposes a structured interface to upstream orchestrators.

Architecture
Pipeline Components

1 · NL-to-Schema Parser

Schema-constrained LLM call converts the user's natural language description into a structured problem specification: geometry type, flow regime, Reynolds number, boundary condition constraints, and output quantities of interest. Structured output prevents hallucinated parameter values from propagating to execution.

2 · Reynolds-Aware Model Selector

Explicit compatibility matrix maps (geometry type, Re range, flow class) to valid (solver, turbulence model) pairs. Covers 12 OpenFOAM solvers and 13 turbulence models. Deterministic selection prevents the model from inventing solver configurations that don't exist in OpenFOAM.

3 · LLM Mesh Generation Engine

Generates blockMeshDict and snappyHexMesh configurations from the structured problem specification. The LLM is prompted with domain-specific grounding about mesh quality requirements (aspect ratio, y+ targeting, refinement zones) for the chosen solver and Re.

4 · OpenFOAM Execution & Validation

Programmatically configures and runs OpenFOAM cases. Post-execution, compares key quantities (drag coefficient, velocity profiles, pressure drop) against literature reference values. Validation gates prevent reporting results that exceed error thresholds without flagging.

Results
Benchmark Performance

100% pass rate across 5 canonical flow regimes

Lid-driven cavity, cylinder flow (Re=100), backward-facing step (BFS), pipe flow, and channel flow — all complete without manual intervention and produce physically plausible results.

Sub-6% error on literature references

Cylinder Re=100 drag coefficient and BFS reattachment length match published DNS and experimental values within 6% — within the range expected from RANS turbulence modeling.

12 solvers × 13 turbulence models covered

Reynolds-aware selector handles incompressible, compressible, and buoyancy-driven flow scenarios. Correct solver-model pairing is critical for physical accuracy and numerical stability.

Significance
Why This Matters

This project sits at the intersection of two distinct domains: large language models and computational fluid dynamics. Most LLM-for-science work either stays at the level of text (summarization, QA) or relies on surrogate models. This agent works with real simulation software — OpenFOAM — making it closer to an autonomous scientific computing assistant than a standard code generator.

The schema-constrained approach and Reynolds-aware selection matrix are deliberate design choices to prevent the LLM from confabulating plausible-sounding but physically incorrect configurations. This is the core challenge in applying LLMs to engineering workflows where errors have real consequences.
