Motivation
Why ML Auditing?

Reproducibility failures in published machine learning research are widespread, yet existing detection tools rely on static code analysis that cannot capture the cross-artifact, sequential reasoning real auditing demands. Auditing an ML experiment requires reading code, results tables, and methodology sections in sequence, forming hypotheses, and citing exact quoted evidence — a fundamentally interactive, multi-step process.

MLAuditBench addresses this gap by framing integrity auditing as a reinforcement learning problem. Rather than asking "can this model classify X?", agents must navigate a structured environment, decide what to inspect, cite exact evidence, and submit a verdict within a fixed step budget — mirroring the actual workflow of a peer reviewer or auditor.

Violation Taxonomy
8 Violation Types (V1–V8)

V1 — Data Leakage

Test-set information leaks into training, pre-processing, or feature selection in a way that inflates reported performance.

V2 — Metric Manipulation

Choosing, redefining, or switching evaluation metrics after observing results to favour a particular conclusion.

V3 — Selective Reporting

Reporting only a subset of experimental conditions, ablations, or runs while omitting results that contradict the narrative.

V4 — Undisclosed Ablations

Running component ablations informally and incorporating findings without disclosing them in the methodology.

V5 — Hyperparameter Post-Hoc Tuning

Tuning hyperparameters on the test set or after observing test performance, then reporting as if tuned on a held-out validation set.

V6 — Baseline Sandbagging

Deliberately undertuning baselines or using inferior baseline implementations to make the proposed method appear stronger.

V7 — Improper Train/Test Split

Violating the statistical independence of training and evaluation sets through temporal leakage, group leakage, or re-use of held-out data.

V8 — Cherry-Picked Seeds

Selecting random seeds that produce favourable results and reporting only those, without disclosing the selection process or variance across seeds.

Benchmark Design
Environment & Reward Structure

The benchmark spans 62 ML experiments across 5 archetypes: classification, regression, generation, RL, and multi-modal. Experiments span three difficulty tiers — Easy, Medium, Hard — with progressively more subtle violation embedding. To prevent gaming, the environment includes adversarially clean episodes (6 of 62), planted red herrings, and a fabricated-evidence penalty.

Reward is structured as: 0.80 × violation_score + 0.10 × efficiency_bonus + 0.10 × verdict_bonus. The efficiency bonus rewards agents that reach correct verdicts with fewer environment steps. The verdict bonus rewards correct binary detection independent of evidence quality. Agents must cite exact quoted text from the artifacts they inspect — paraphrase is penalised.

Nine inference-script failure modes were catalogued during environment construction, including variance collapse (σ = 0.39 → 0.00 across seeds), silent metric switching, and off-by-one index errors. These failure modes seed the V1–V8 violation instances with realistic, naturally-occurring bugs rather than synthetic injections.

Results
Model Comparison (Easy / Medium / Hard)

GPT-4.1-mini v2 — 0.900 / 0.901 / 0.819

Matches or surpasses the human baseline on every tier. Best automated system overall. Strong evidence-citation behaviour observed on V1 and V7 violations.

GPT-4.1 v2 — 0.900 / 0.783 / 0.780

Comparable to GPT-4.1-mini on Easy but degrades more steeply on Hard. Larger context window does not translate to improved hard-tier performance.

o4-mini v2 — 0.539 / 0.588 / 0.469

Suffers systematic hallucination failures — fabricates evidence citations that don't appear in the artifacts. Below human baseline on all tiers.

Human Baseline (10-seed) — 0.82 / 0.86 / 0.57

ML researchers with domain background. Humans excel on Medium but drop significantly on Hard compound violations requiring cross-artifact reasoning.

Claude Sonnet 4.6 Interactive — 0.954 ceiling (Hard compound)

Expert interactive audit session establishes the ceiling. All automated agents fall below this, confirming the benchmark discriminates at frontier capability levels.

Infrastructure
OpenEnv-Compliant Docker Release

MLAuditBench is released as an OpenEnv-compliant Docker container with fixed seeds, full per-seed logs, and a public human-evaluation kit. The container exposes a standard gym-compatible interface: reset() returns a structured experiment bundle; step(action) accepts artifact inspection actions or verdict submission; render() returns a human-readable audit trail.

All 62 experiments are versioned with content-addressed artifact hashes to prevent benchmark contamination. The human evaluation kit provides a browser-based interface identical to the agent environment, enabling direct human-vs-agent comparison under identical conditions.
