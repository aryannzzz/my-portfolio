Beyond Static RAG: How Temporal Intelligence Changes Financial Information Retrieval

Inter IIT Tech Meet | Pathway Problem Statement

When Yesterday's Truth Becomes Today's Misinformation

"What's Apple's current revenue guidance?" A financial analyst asks this question on November 14, 2024. A standard RAG system retrieves Apple's 10-K filing from fiscal year 2023, which states revenue guidance of $380-395 billion. The analyst makes investment decisions based on this answer.

There's just one problem: Apple revised its guidance in an October 2024 earnings call to $410-420 billion, a 10% increase. The 10-K filing is authoritative and well-documented, but it's wrong. The earnings call transcript is recent and correct, but it's buried in a sea of financial news.

Traditional RAG systems don't understand time. They retrieve by semantic similarity, not temporal relevance. In finance, this can be catastrophic. Markets move on new information. Yesterday's data, no matter how authoritative, can be dangerously misleading.

For our Inter IIT project, I built a Hybrid RAG system that doesn't just retrieve information, it understands temporal relationships, detects contradictions between old and new sources, and weighs recency against authority.

The Problem: Static Knowledge in Dynamic Markets

Why Standard RAG Fails in Finance

  • No Temporal Awareness: Treats a 2-year-old 10-K the same as yesterday's news
  • No Contradiction Detection: Doesn't notice when new information conflicts with old
  • No Recency Weighting: High-quality old information ranks above medium-quality new information
  • No Authority Distinction: Treats official SEC filings and random tweets equally

Financial analysis requires understanding that:

  • SEC filings are authoritative but infrequent (quarterly/annual)
  • Earnings calls provide updates but lack detail
  • News articles are timely but may lack accuracy
  • Social media is real-time but full of noise

The challenge: build a system that balances recency with authority, detects when new information supersedes old, and handles contradictions intelligently.

The Solution: Dual-Index Temporal RAG

I designed a hybrid architecture with two complementary knowledge bases:

Static Index: Authoritative Historical Knowledge

  • Content: SEC 10-K/10-Q filings, annual reports, audited financials
  • Update Frequency: Quarterly/annual
  • Authority Level: Very high (legal documents)
  • Temporal Span: Years of historical data

Live Index: Real-Time Market Intelligence

  • Content: News articles, earnings call transcripts, analyst reports, social sentiment
  • Update Frequency: Real-time streaming
  • Authority Level: Variable (source-dependent)
  • Temporal Span: Rolling 90-day window
class HybridRAGSystem:
    def __init__(self):
        # Authoritative but infrequent
        self.static_index = StaticDocumentIndex()  # 10-K filings
        
        # Real-time but noisy
        self.live_index = StreamingNewsIndex()     # News/updates
        
        # Temporal relevance calculator
        self.temporal_model = TemporalRelevanceModel()

Temporal Weighting: When Is Information Valid?

Not all information has the same temporal validity period:

  • Quarterly Revenue: Valid for 3 months, then superseded
  • CEO Change: Event happens once, permanent effect
  • Market Sentiment: Valid for hours/days, highly volatile
  • Business Model: Valid for years, slowly evolving

Exponential Temporal Decay

News items lose relevance over time. I implemented exponential decay with topic-specific half-lives: 

class StreamingNewsIndex:
    def __init__(self):
        self.temporal_decay_factor = 0.95  # Decay per 24 hours
        
        # Topic-specific half-lives
        self.half_lives = {
            'earnings': 90,      # Quarterly validity
            'sentiment': 7,      # Weekly relevance
            'leadership': 365,   # Yearly impact
            'breaking_news': 1   # Daily freshness
        }
    
    def compute_temporal_weight(self, doc, topic):
        age_days = (datetime.now() - doc.timestamp).days
        half_life = self.half_lives.get(topic, 30)  # Default 30 days
        
        # Exponential decay: weight = 0.5^(age/half_life)
        weight = 0.5 ** (age_days / half_life)
        
        return weight

Contradiction Detection: When Sources Disagree

The most challenging problem: what happens when the 10-K says one thing and recent news says another?

Example Contradiction

Static Index (2023 10-K):
"Apple's revenue guidance for FY2024 is $380-395 billion"
Authority: 10/10 | Recency: 2/10

Live Index (Oct 2024 News):
"Apple raises FY2024 guidance to $410-420 billion"
Authority: 7/10 | Recency: 10/10

CONTRADICTION DETECTED

Resolution Strategy

def resolve_contradiction(static_claim, live_claim):
    """Intelligent conflict resolution"""
    
    # Calculate confidence scores
    static_confidence = (
        static_claim.source_authority * 0.3 +  # Authority weight
        static_claim.verification_score * 0.5 + # Verification
        static_claim.recency_score * 0.2        # Recency (low for old data)
    )
    
    live_confidence = (
        live_claim.recency_score * 0.7 +        # High recency weight
        live_claim.source_credibility * 0.2 +   # Medium authority
        live_claim.corroboration_count * 0.1    # Multiple sources?
    )
    
    # Resolution logic
    if live_confidence > static_confidence + 0.2:
        return ResolutionType.SUPERSEDED
    elif static_confidence > live_confidence + 0.2:
        return ResolutionType.AUTHORITATIVE
    else:
        return ResolutionType.UNCERTAIN

Resolution Types and Responses

  • SUPERSEDED: "Recent guidance (Oct 2024) updated to $410-420B, superseding previous $380-395B guidance"
  • AUTHORITATIVE: "Official filing remains authoritative despite recent speculation"
  • UNCERTAIN: "Mixed signals: filing suggests X while recent reports indicate Y"

The Retrieval and Synthesis Pipeline

Step 1: Dual Retrieval

def synthesize_answer(query, symbol='AAPL'):
    # Retrieve from both sources
    static_docs = self.static_index.search(query, k=5)
    live_news = self.live_index.search(query, k=10)
    
    # Extract key claims from each
    static_claims = extract_claims(static_docs)
    live_claims = extract_claims(live_news)

Step 2: Temporal Relevance Scoring

    # Score each document by temporal relevance
    temporal_scores = {}
    
    for doc in static_docs:
        age = (datetime.now() - doc.filing_date).days
        # Static docs are less sensitive to age (quarterly data)
        temporal_scores[doc.id] = 1.0 if age < 180 else 0.7
    
    for doc in live_news:
        age_hours = (datetime.now() - doc.timestamp).total_seconds() / 3600
        # Live news decays exponentially
        temporal_scores[doc.id] = 0.95 ** (age_hours / 24)

Step 3: Contradiction Detection

    # Find contradictions between sources
    contradictions = []
    
    for static_claim in static_claims:
        for live_claim in live_claims:
            similarity = semantic_similarity(static_claim, live_claim)
            
            # High similarity but different values = contradiction
            if similarity > 0.8 and not values_match(static_claim, live_claim):
                contradictions.append({
                    'static': static_claim,
                    'live': live_claim,
                    'similarity': similarity
                })

Step 4: Conflict Resolution

    resolutions = []
    
    if contradictions:
        for conflict in contradictions:
            resolution = self.resolve_conflict(
                conflict['static'],
                conflict['live'],
                recency_weight=0.7,    # Favor recent info
                authority_weight=0.3   # But respect authority
            )
            resolutions.append(resolution)

Step 5: Context Building and Response Generation

    # Build final context with temporal awareness
    context = self.build_temporal_context(
        static_docs=static_docs,
        live_news=live_news,
        temporal_scores=temporal_scores,
        resolutions=resolutions
    )
    
    # Generate response that acknowledges time
    response = self.generate_response(query, context)
    
    return response

Advanced Features: Entity-Centric Indexing

Financial queries often involve multiple entities (companies, executives, products). The system tracks entity mentions and their temporal relationships. 

class EntityTemporalGraph:
    """Track how entity relationships evolve over time"""
    
    def add_relationship(self, entity1, entity2, relationship, timestamp):
        self.graph.add_edge(
            entity1,
            entity2,
            relationship=relationship,
            valid_from=timestamp,
            valid_until=None  # Still valid
        )
    
    def invalidate_relationship(self, entity1, entity2, timestamp):
        """Mark relationship as no longer valid"""
        edge = self.graph.get_edge(entity1, entity2)
        if edge:
            edge.valid_until = timestamp

# Example: CEO changes
graph.add_relationship("Apple", "Tim Cook", "CEO", "2011-08-24")
# This relationship is still valid today

graph.add_relationship("Twitter", "Jack Dorsey", "CEO", "2015-10-05")
graph.invalidate_relationship("Twitter", "Jack Dorsey", "2021-11-29")
graph.add_relationship("Twitter", "Parag Agrawal", "CEO", "2021-11-29")
# Historical relationships preserved with temporal bounds

Different types of financial information update at different frequencies:

Information Type Update Frequency Validity Period Authority Level
10-K Filing Annual 1 year Very High
10-Q Filing Quarterly 3 months Very High
Earnings Call Quarterly 3 months High
Press Release Ad-hoc Until next update Medium-High
News Article Continuous Days to weeks Medium
Social Media Real-time Hours Low

The system uses this hierarchy to determine which source to trust when contradictions arise.

Performance Optimization: Incremental Updates

Re-indexing all documents every time new information arrives is expensive. The system uses incremental updates: 

class IncrementalIndex:
    def update_with_new_document(self, doc):
        # Extract entities and claims
        entities = self.extract_entities(doc)
        claims = self.extract_claims(doc)
        
        # Check for contradictions with existing knowledge
        for claim in claims:
            contradicting_claims = self.find_contradictions(claim)
            
            if contradicting_claims:
                # Mark old claims as potentially superseded
                for old_claim in contradicting_claims:
                    old_claim.superseded_by = doc.id
                    old_claim.superseded_at = doc.timestamp
                
                # Create versioned knowledge
                self.knowledge_graph.add_version(
                    claim=claim,
                    supersedes=[c.id for c in contradicting_claims],
                    valid_from=doc.timestamp
                )
        
        # Add to searchable index
        self.vector_index.add(
            embedding=self.embed(doc),
            metadata={
                'timestamp': doc.timestamp,
                'entities': entities,
                'authority_score': doc.authority,
                'source_type': doc.source_type
            }
        )

The Future: From RAG to True Continual Learning

Temporal RAG is powerful, but it's still fundamentally about retrieval. The system doesn't truly "learn" from new information, it just fetches it at query time.

RAG Limitations

  • No Weight Updates: Model parameters stay frozen
  • Context Window Constraints: Limited by token limits
  • Retrieval Overhead: Every query requires vector search
  • No Pattern Internalization: Can't learn meta-patterns across documents

The Dragon Hatchling Paradigm

Recent research (like the Dragon Hatchling paper) proposes true continual learning where models update their internal weights as new information arrives, without forgetting old knowledge.

Key Differences

Aspect RAG (Current) Continual Learning (Future)
Learning Simulated via retrieval Real weight updates
Memory External vector database Internal differentiable memory
Inference Retrieve then generate Direct generation from learned patterns
Latency Higher (retrieval overhead) Lower (no retrieval needed)

Temporal RAG is a bridge technology. It provides the temporal intelligence that standard RAG lacks, while paving the way for truly adaptive models that internalize knowledge over time.

Lessons Learned: Building Temporal AI Systems

1. Time Is a First-Class Feature

Timestamps can't be metadata. They must be core to the retrieval and ranking logic.

2. Authority and Recency Are Both Essential

You can't just favor recent information. A well-sourced old document beats a sketchy new rumor.

3. Contradictions Are Signals, Not Errors

When sources disagree, that's valuable information. The system should surface contradictions, not hide them.

4. Different Information Has Different Half-Lives

Quarterly earnings decay faster than business model descriptions. Topic-aware temporal weights are crucial.

5. Versioned Knowledge Is Essential

Don't just overwrite old information. Maintain historical versions with temporal validity bounds.

The Bottom Line: Static RAG systems retrieve information. Temporal RAG systems understand time. By combining authoritative historical knowledge with real-time updates, detecting contradictions between sources, and intelligently weighting recency against authority, we built a system that doesn't just find information, it understands when that information is valid. This is the difference between finding the right document and providing the right answer.
